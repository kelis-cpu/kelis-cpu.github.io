# 一。业务逻辑漏洞

业务逻辑的主要目的是强制执行在设计应用程序或功能时定义的规则和约束，包括防止用户会做业务产生负面影响的事情。业务逻辑漏洞是应用程序设计和实现中的缺陷，允许攻击者引发意外行为，使攻击者能够操作合法功能以实现恶意目标，这些缺陷通常是由未能预测可能发生的异常应用程序状态。

识别业务逻辑漏洞需要一定量的人类知识，例如对业务领域的理解或攻击者在给定上下文中可能具有的目标，使得难以使用自动漏洞扫描程序进行检测。

业务逻辑漏洞怎样产生：

设计和开发团队对用户如何与程序交互做出了错误的假设，例如，开发人员假设用户仅通过 Web 浏览器传递数据，则很可能被使用拦截代理的攻击者绕过。逻辑缺陷在开发团队自己都不完全理解的复杂的洗系统中尤为常见，开发人员无法深入了解应用程序的所有领域是如何工作的，在一个组件上工作的人会对另一个组件的工作方式做出缺陷的假设。

业务逻辑漏洞的影响：

逻辑缺陷的影响取决于它所涉及的功能，例如，如果存在于身份验证机制，则可能会对受害者整体的安全性产生严重影响，攻击者可以利用它来提权，或完全绕开身份验证，获得对敏感数据和功能的访问权限；金融交易中的错误逻辑显然会通过被盗资金、欺诈等给企业造成巨大损失，即使不允许攻击者直接受益，但是也能以某种方式破坏业务。

# 二。常见的业务逻辑漏洞

# 1. 过度信任客户端

根本有缺陷的假设是用户只会通过提供的 Web 界面与应用程序交互。

# 2. 无法处理非常规输入

应用程序可能被设计为接受某种数据类型的任意值，但从业务角度，逻辑决定了这个值是否可以接受，包括管理库存，应用预算限制，触发供应链阶段等限制，如转账 - 1000 元，在审核应用程序时，可以使用 Repeater 来提交异常高，异常低或基于文本字段的异常长字符串。假设目标网站发现一个表单无法安全处理非常规输入，那么其他表单很可能会遇到同样问题。

# 3. 受信任的用户并不总是值得信任

整个应用程序没有一致地应用业务规则和安全措施。

# 4. 用户不会总是提供强制输入

浏览器可能会阻止普通用户在没有必要输入的情况下提交表单，但攻击者可以在传输过程中篡改参数，甚至扩展到完全删除参数。在同一个服务端脚本中实现多个功能的情况下，特定参数的存在或不存在可以确定执行哪个代码，删除参数值可能允许攻击者访问本无法访问的额代码路径。

检查逻辑漏洞时，应尝试一次删除每个参数，并观察其对响应的影响。

# 5. 用户不会总是遵循预期的顺序

许多事物依赖于由一系列步骤组成的预定义工作流。比如网站要求用户在一个页面登录，然后再另一个页面输入验证码，假设不验证用户是否遵循此过程这样做，可能会允许攻击者完全绕过 2FA 步骤。

这种情况通常会导致异常，因为预期变量具有 null 或未初始化的值，到达处于部分定义或不一致状态的位置也可能导致应用程序出错，应该多注意任何错误信息或调试信息。

# 6. 特定领域的漏洞

在线商店的折扣功能是寻找逻辑缺陷的典型攻击面，例如，一个商店为超过 100 元的订单提供 10% 的折扣，如果业务逻辑在应用折扣后无法检查订单是否有效，则攻击者可以将商品添加到购物车，直到到达 100 元的门槛，然后在下订单之前删除不要的商品，然会即使订单不满足预期标准也会收到折扣。

# 7. 加密预言机

当用户可控输入被加密且生成的密文随后以某种方式提供给用户时，可能会出现漏洞。攻击者可以使用相同的加密算法和非对称密钥加密任意数据，当应用程序中 2 有其他用户可控输入期望使用相同加密算法的数据时，攻击者可能会使用加密预言机生成有效的加密输入，然后传递给其他敏感函数。

如果网站有一个提供反向功能的用户可控输入，这将使攻击者能够解密其他数据以识别预期的结构。

# 三。如何减少业务逻辑漏洞

防止业务逻辑漏洞的关键：

  • 确保开发和测试人员了解应用程序服务的领域，避免对用户行为或应用程序其他部分做出隐含的假设。
  • 确定对服务器状态所作的假设，并实施必要的逻辑来验证这些假设是否满足，包括任何输入值都是合理的。

开发人员遵循：

  • 维护清晰的设计文档和数据流。
  • 尽可能清楚编写代码，写好注释，注意代码风格。
  • 注意对使用每个组件的其他代码的应用，攻击者可能以不寻常的方式操纵它们。
