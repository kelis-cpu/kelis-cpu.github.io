# 一。身份验证

身份验证是验证给定用户或客户端身份的过程。

三个身份验证因素：

  • 知识因素。知道的信息，如密码 / 安全问题。
  • 占有因素。拥有的东西，如手机 / 安全令牌等物理对象。
  • 内在因素。生物特征、行为模式。

身份验证依赖于一系列技术来验证这些因素中的一个或多个。

身份验证和授权的区别：

身份验证是验证用户是否真的是他们声称的身份的过程，而授权涉及验证用户是否被允许做某事。

# 二。身份验证漏洞

身份验证机制中的大多数漏洞出现在一下两种方式中：

  • 身份验证机制较弱，被暴力破解。
  • 逻辑缺陷或糟糕的编码导致攻击者绕过身份验证机制。

身份验证漏洞的影响：

一旦攻击者绕过身份验证或暴力破解一个账户，就可以访问账户所拥有的所有数据和功能，如果能进行提权，还可以控制整个应用程序，并可能获得对内部基础设施的访问权限。

# 三。常见的身份验证漏洞

网站的身份验证系统通常由几种不同的机制组成。

# 1. 基于密码登录中的漏洞

# a. 暴力破解攻击

1) 暴力破解用户名

  • 用户名有一定格式

例如邮箱，查看业务登录时很常见 (firstname.lastname@somecompany.com)，即使没有明显格式，也可以猜测 admin、administrator 等管理员账户。

  • 检查网站是否有公开披露的用户名
  • 是否可以在不登录的情况下访问用户配置文件

即使配置文件的实际内容被隐藏，而配置文件中使用的名称有时与登录用户名相同。

  • 检查 HTTP 响应中是否有任何邮箱泄露、响应宝中是否包含管理员和运维的邮箱。

2) 暴力破解密码

破解密码和破解用户名类似，但是难点在于确定密码的长度，且网站要求用户使用高熵值密码 -- 最小长度、大小写混合、至少使用一个特殊符号，但是可以利用人类行为进行猜测，用户通常不会使用随机字符来创建强密码，例如 mypwd 不允许，用户会尝试 mypwd1 或 my1wd 代替；在定期要求用户更改 密码的情况下，用户通常只会进行较小的、可预测的更改。

3) 用户名枚举

用户名枚举通常发生在攻击者可以看见用户名是否合法的变化的场景下，比如登录时，输入正确用户名但是密码不对、注册用户名已存在。

尝试在登录页暴力破解时，多注意以下地方：

  • 状态码

暴力破解期间，返回的 http 状态码绝大多数都是相同的，因为大多数猜测都是错误的。无论结果如何让应该返回相同的状态码，但网站不总是遵循这种做法。

  • 错误信息

有时返回得错误消息会有所不同，具体取决于用户名和密码都不正确或只有密码不正确。

  • 响应时间

多数请求都以相似的响应时间处理，任何与此不同的请求服务器可能都有不同的处理过程，网站可能仅在用户名有效时才检查密码是否正确，这个额外的步骤会导致响应时间有增加，攻击者可以使用一个过长的密码来使这种延迟更明显。

# b. 有缺陷的暴力破解保护

防止暴力破解的两种方法：

  • 登陆失败次数过多，锁定尝试访问的账户
  • 连续快速登录次数过多，ban IP

漏洞：如果多次登陆失败，有时可能被 ban IP，在一些措施中，如果 IP 所有者成功登录，则失败尝试次数的计数器会重置，这就意味着攻击者只需每隔几次尝试登录自己的账户则可防止达到此限制。如果服务器端支持 X-Forwarded-For (表示请求端真实 IP) 头，则可以绕过 ip 地址暴力破解保护。

1）账户锁定

如果一个用户登录失败次数过多，网站可能会锁定该账号，攻击者可以利用锁定信息来枚举存在的账号。

2）用户速率限制

由于不容易受到用户名枚举和拒绝服务攻击，用户速率限制优于账户锁定。但是它仍不安全，可以通过绕过 IP 地址；IP 锁定一般三种方法解锁：时间到自动解锁；管理员解锁；验证码

由于该限制基于用户 IP 地址发送 HTTP 请求的速率，如果清楚如何通过单个请求猜测多个密码，也可以绕过该防御。加入密码通过 json 格式传送，可以采用 json 数组来传递多个密码，

# c.HTTP 基本认证

在 HTTP 基本认证中，客户端从服务器接受一个身份验证令牌，该令牌由用户名和密码连接起来，并使用 base64 编码，此令牌由浏览器存储和管理，浏览器会自动添加到后续每个请求的 Authorization 中，

Authorization:bas64(username:pwd)

但容易受到 csrf 和暴力破解攻击，某些情况下 HTTP 基本身份验证只会授予攻击者看似无趣的页面的权限。

# 2. 多因素身份验证中的漏洞

# a. 两因素身份验证令牌

RSA 令牌或专业移动 APP-Google Authenticator。

一些网站将验证码作为短信发送给手机，可能会被拦截，攻击者还可以通过七杂的手段获取受害者电话号码的 SIM 卡，然后收到发送给受害者的所有 SMS 信息。

# b. 绕过两因素身份验证

如果网站首先提示输入密码，再在单独的额页面提示输入验证码，则用户在输入验证码之前实际上处于 “登录” 状态，在这种情况下，可以尝试跳过第二步验证，直接访问其他页面。

# c. 有缺陷的两因素验证逻辑

双重身份验证中的逻辑缺陷意味着在用户完成初始登录步骤后，网站无法充分验证同一用户是否正在完成第二步。

# d. 暴力破解 2FA 验证码

如果用户输入了一定数量的错误验证码，一些网站会尝试通过自动注销账户来防止暴力破解，但是可以通过 Intruder 创建宏来自动化多步骤操作。

# 3. 其他身份验证机制漏洞

# a. 记住登录

一些网站的记住登录依靠持久化 cookie 实现，而这些 cookie 可能采用固定的格式，如用户名 + 时间戳，甚至会加上密码。如果网站使用开源框架，cookie 的结构直接可以公开查询，也可以通过 xss 盗取用户 cookie，cookie 需要经过加盐的 hash 才能保证安全。

# b. 重置密码

  • 通过邮箱发送密码

容易遭受中间人攻击，安全性取决于生成的密码在很短时间内过期，或者用户立即再次更改密码。

  • 通过网址重置密码

网址很容易被猜出是哪个用户在重置密码；验证 token 失败，导致攻击者直接使用自己的重置页面，删去 token 值就绕过验证，重置任意用户密码.

如果网址是动态生成的，也可能遭到密码重置中毒攻击，在这种情况下，攻击者可能窃取另外一个用户的令牌来更改他们的密码。

  • 修改用户密码

修改密码一般需要提供当前密码和新密码，网站和正常登录页一样会验证用户名和当前密码是否正确，因此修改用户密码会受到登录页相同的攻击。如果允许攻击者不登录，直接修改密码，比如在隐藏字段提供用户名。攻击者可能在请求中编辑此值来针对任意用户，可能被利用来枚举用户名和暴力破击密码。

# 四。密码重置中毒

# 1. 密码重置

最通用的密码重置策略：

  1. 用户输入用户名或邮箱，提交密码重置请求。
  2. 网站验证用户是否存在，生成一个临时、唯一、高熵与用户账号关联的 Token。
  3. 网站将包含 Token 的网址发送给用户邮箱。
  4. 用户访问网址，重置密码，网站销毁 Token。

该方法安全性依赖于只有目标用户才能访问收件邮箱，密码重置中毒就是一种窃取此 Token 以更改其他用户密码的方法。

# 2. 如何构造密码重置中毒攻击

如果发送给用户的网址是基于可控输入生成的，比如 Host/X-Forwarded-Host 头，那么可以按如下步骤构造攻击：

  1. 攻击者获取受害者的电子邮件或用户名，并代表他们提交密码重置请求，提交时，拦截生成的 HTTP 请求并修改 Host 头使其指向受攻击者控制的域，比如， evil-user.net
  2. 受害者从网站接收到一封真正的密码重置邮件，包含了一个有效密码重置 Token，但其中的 URL 中的域名指向了攻击者的服务器， https://evil-user.net/reset?token=0a1b2c3d4e5f6g7h8i9j
  3. 受害者点击该网址，或其他方式获取该连接，例如通过防病毒扫描程序，密码重置 Token 就会发送带攻击者服务器。

真实的攻击在，攻击者可能还会提前发送一些虚假的违规通知来增加受害者点击的可能。

# 五。如何确保身份验证机制安全

  • 注意用户凭据

登录请求都是与 HTTPS，其它 HTTP 请求也重定向到 HTTPS；审核网站，确保没有用户名或邮箱通过公开的配置文件或在 HTTP 响应中泄露。

  • 不要指望用户

Dropbox 开发的 js 库 zxcvbn 来检查密码的安全性。

  • 防止用户名枚举

无论用户名是否有效，都应该使用通用错误消息；每个登录请求返回相同的状态码，响应时间尽可能难以区分。

  • 实施强大的蛮力保护措施

基于 IP 的用户速率限制，要求用户在达到特定限制后每次登录尝试都完成验证码测试。

  • 检查验证逻辑
  • 不要忘记补充功能

密码重置或更改与主要登陆机制一样是有效的攻击面。

  • 实施适当的多因素身份验证机制

注意 2FA 的逻辑检查。
