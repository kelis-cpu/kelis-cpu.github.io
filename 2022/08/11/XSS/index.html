# 三种类型的 XSS

# 一。反射型 XSS

当应用程序接受到 HTTP 请求中的数据并以不安全的方式将这些数据包含在即时响应中，就会出现反射 XSS。

假设一个网站有一个搜索功能，

https://insecure-website.com/search?term=gift

应用对该 URL 的响应中提供了搜索词， <p>You searched for: gift</p>

程序如果不执行任何数据处理，攻击者可以伪造如下攻击

https://insecure-website.com/search?term=<script>/*+Bad+stuff+here...+*/</script>

此 URL 响应 <p>You searched for: <script>/* Bad stuff here... */</script></p> , 如果应用程序的一个用户请求攻击者的 URL，那么攻击者的脚本就会在受害者用户的浏览器中执行。

反射 xss 的影响：

  • 执行用户可以执行的操作
  • 查看和修改用户可以查看和修改的任何信息
  • 发起与其他应用程序用户的交互，包括恶意攻击。

攻击者诱使用户发出请求的方式包括将链接放置在攻击者控制的网站上，或放在允许生成内容的另一个网站，或通过电子邮件、推送或其他消息中发送链接。

# 探测反射 xss 的方法

最快的是使用 burp，手工探测步骤如下：

  • 测试每个 HTTP 请求包含的参数

分别测试 HTTP 请求中数据的每个入口点，包括 URL 查询字符串、消息正文中的参数、URL 文件路径、HTTP 头。

  • 提交随机字母数字值

对于每个入口点，提交一个唯一的随机值并确定该值是否反映在响应中，可以利用 Intruder 随机生成十六进制值来生成合适的随机值，再利用 grep payloads 自动标记包含提交值得响应。

  • 确定反射发生的位置

可能发生在 HTML 标记之间的文本中、被引用的标记属性中、JS 字符串中等。

  • 测试有效 payload

将原始随机值留在请求中，并将测试 payload 放在其之前或之后。

  • 最后在浏览器中测试攻击

反射 xss 和存储 xss 的区别：

当应用程序从 HTTP 请求中获取一些输入并以不安全的方式将该输入输出嵌入到即时响应中，就会发生反射 xss，而存储 xss 则是应用程序将这些输入存储并以不安全的方式将其嵌入到稍后的响应中。

反射 xss 和 self-xss 的区别：

两者的应用程序行为相似，但是 self-xss 不能通过精心设计的 URL 或跨域请求以正常的方式触发，只有当受害者自己从浏览器提交 xss payload 时，才会触发 self-xss，通常用来进行社工。

# 二。存储型 XSS

当应用程序从不受信任的来源接收数据并以不安全的方式将该数据包含在其以后的 HTTP 响应中时，就会出现存储 xss。

# 探测存储 xss 的方法

手工测试存储 XSS 难度很大，需要测试所有相关的入口点 - 攻击者可控的数据，以及该数据可能出现在应用程序响应中的所有出口点。

入口点包括：URL 查询字符串和消息正文中的参数或其他数据；URL 文件路径；攻击者可以通过其将数据传递到应用程序的任何路由，存在路由取决于应用程序实现的功能：邮件程序将从处理电子邮件中收到数据；显示推送的应用程序可能会处理第三方推送中包含的数据。

出口点是任何情况下返回给任何类型的应用程序的所有可能的 HTTP 响应。

当找到了出入点和出入点之间的连接后，接下来的测试方法同反射 xss。

# 三.DOM 型 XSS

DOM-Document Object Model，文档类型对象，可以理解为网页的 API，它将网页中的各个元素都看作对象，从而使网页中的元素都可以被获取或编辑。

基于 DOM 的 XSS 通常出现在 JS 从攻击者可控的源，如 URL 获取数据并将其传递到支持动态代码执行的接收器，如 eval () 或 innerHTML，这使攻击者可以执行恶意 JS。

# 探测 DOM 型 XSS

1. 测试 HTML Sinks

将随机字符串放入源中，如 location.search，然后使用开发人员工具检查 HTML 并查找字符串出现的位置，不能使用 “查看源代码” 选项，它没有考虑 JS 在 HTML 中执行的更改。如果字符串出现在双引号属性中，可以尝试在字符串中插入双引号形成闭合。

2. 测试 JS 执行 Sinks

使用它来测试，输入不一定会出现在 DOM 中的任何位置，需要使用 JS 调试器来确定输入是否 / 如何发送到 Sinks。对于每个潜在的源，如 location，首先需要在 JS 代码中找到引用源的例子，找到读取源的位置之后，可以使用 JS 调试器添加断点并跟踪源值的使用方式。源可能被分配给其他变量，这种情况需要再次使用搜索功能查看它们是否被传递到 Sinks。

当发现一个 Sinks 被分配了源的数据时，可以使用调试器将鼠标停在变量上来检查，然后与 HTML Sinks 一样，需要细化输入以查看是否可以进行 XSS 攻击。

3.DOM Invader 测试

利用 Burp 浏览器内置的 DOM Invader 扩展进行测试。

# XSS 可以完成的事：

冒充受害者用户、执行用户能够执行的任何操作、读取用户能够读取的任何数据、捕获用户的登录凭据、对网站进行污染、注入木马。

# 探测 XSS

多数的 XSS 漏洞都能被 Burp 扫描到，

手工测试反射和存储 XSS：提交随机字符串到每个入口点，找到在 HTTP 响应中所提交的字符串的每个位置，再单独测试每个位置以确定是否可以执行 js；

手工测试 DOM 型 XSS：在提交随机字符串之后使用开发人员工具在 DOM 中搜索，再测试是否可利用，但是基于 doucument.cookie/setTimeout 的无法通过审查 js 代码发现，可以使用 Burp 自动检测基于 DOM 的 XSS。

# CSP - 内容安全策略

CSP（Content Security Policy）即内容安全策略，为了缓解很大一部分潜在的跨站脚本问题，浏览器的扩展程序系统引入了内容安全策略（CSP）的一般概念。这将引入一些相当严格的策略，会使扩展程序在默认情况下更加安全，开发者可以创建并强制应用一些规则，管理网站允许加载的内容。
CSP 的实质就是白名单机制，对网站加载或执行的资源进行安全策略的控制。

# CSP 语法

CSP 中常见的 header 字段为 Content-Security-Policy。

一个 CSP 头由多组 CSP 策略组成，中间由分号分隔，如下：

Content-Security-Policy: default-src 'self' www.baidu.com; script-src 'unsafe-inline'

其中每一组策略包含一个策略指令和一个内容源列表。

# 策略指令

# default-src

default-src 作为所有其他指令的备用，一般来说 default-src 'none'; script-src 'self' 这样的情况就会是 script-src 遵循 self，其他的都会使用 none。也就是说，除了被设置的指令以外，其余指令都会被设置为 default-src 指令所设置的属性。

# script-src

script-src 指令限制了所有 js 脚本可以被执行的地方，包括通过链接方式加载的脚本 url 以及所有内联脚本，甚至包括各种方式的引用。其中还有一个很重要的参数叫 'unsafe-inline' ，如果加上这个参数，就不会阻止内联脚本，但这被认为是不安全的。

对于这个属性有个特殊的配置叫 unsafe-eval ，它会允许下面几个函数：

eval() Function() setTimeout() with an initial argument which is not callable.setInterval() with an initial argument which is not callable.

# 关键字

'none'
代表空集；即不匹配任何 URL。两侧单引号是必须的。
'self'
代表和文档同源，包括相同的 URL 协议和端口号。两侧单引号是必须的。
'unsafe-inline'
允许使用内联资源，如内联的 script 元素、 javascript: URL 、内联的事件处理函数和内联的 style 元素，两侧单引号是必须的。
'unsafe-eval'
允许使用 eval() 等通过字符串创建代码的方法。两侧单引号是必须的。

# 数据

data :
允许 data: URI 作为内容来源。

mediastream :
允许 mediastream: URI 作为内容来源。

  1. csp 指令可以指定一个随机数，并且在加载脚本的标签中必须使用相同的值。
  2. csp 指令指定受信任脚本内容的哈希值。

# AngualrJS 沙箱

该沙箱可防止在 AngularJS 模板表达式中访问潜在危险对象，如 document 或 window，还可以阻止访问具有潜在为危险的属性，如__proto__，虽然在 1.6 版本删除了沙箱功能，但是许多应用程序仍然在使用旧版本的 AngularJS.

沙箱工作原理

解析表达式，重写 js 代码，然后使用 各种函数预测重写的代码是否包含任何危险对象，如 ensureSafeObject() 函数检测给定对象是否引用自身，也是一种检测 window 对象的方法，通过检查构造函数属性是否引用自身检查 Function 构造函数。

ensureSafeMemberName() 函数检查对象的各个属性访问，如果包含 __proto__ 或 __lookupGetter__ 等危险属性，该对象将会被阻止访问， ensureSafeFunction() 函数可防止调用 call()、apply()、bind() 或 constructor() 。

可以通过 http://jsfiddle.net/2zs2yv7o/1 / 查看沙箱运行情况。

沙箱逃逸就是诱使沙箱认为恶意表达式是良性的。

# 一些绕过技巧

如果网站不能使用单 / 双引号，可以使用字符串的构造函数 toString() 来生成字符，标准的沙箱逃逸中可以使用 $eval () 来执行 js 有效载荷，也可以使用 orderBy 过滤器，语法为： [123]|orderBy:'some string' , | 在 Angularjs 中表示过滤操作，该表达式将左侧的数组 [123] 发送到右侧的 orderBy 过滤器，冒号表示要发送到过滤器的参数 - 一个字符串，orderBy 过滤器通常用于对对象进行排序，但也接受表达式，这就意味着可以传递有效 payload。

csp 中沙箱逃逸

csp 将阻止 js 事件，Angularjs 定义了一个特殊的 $event 对象来简单引用浏览器事件对象，该对象有一个特殊的属性 path ，该属性包含导致事件执行的对象数组，最后一个属性始终是 window 对象，可以使用它来执行沙箱逃逸。

<input autofocus ng-focus="$event.path|orderBy:'[].constructor.from([1],alert)'">

from() 函数将对象转换为数组并在该数组的每个元素上调用给定函数 (在第二个参数中指定)。如果有长度限制，可以使用以下 payload： [1].map(alert) , map() 函数接受一个函数作为参数，并将为数组中的每个元素调用它。

阻止 AngularJS 注入的方法是不将用户的输入直接生成模板或表达式

# 防止 XSS

  1. 接收用户输入时，尽可能严格地根据预期或有效输入进行过滤。
  2. 在 HTTP 响应中输出用户可控数据时进行编码以防止被解释为活动内容。
  3. 使用适当的响应头，Content-Type/X-Content-Type-Options 确保浏览器以想要的方式解释响应。
  4. CSP

PHP 防止 XSS：使用字符白名单过滤输入，使用类型提示或类型转换，使用 HTML 上下文的 htmlentities 或 ENT_QUOTES 或 js 的 unicode 转义输出。

Java 防止 XSS：使用字符白名单过滤输入，使用如 Google 之类的库对 HTML 上下文的输出进行 HTML 编码，或 js 的 Unicode 转义。
