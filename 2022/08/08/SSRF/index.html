# 一。常见的 SSRF 攻击

服务端请求伪造允许攻击者诱导服务器端应用程序向非预期位置发出请求，典型的 SSRF 攻击中，攻击者可能会导致服务器与组织基础架构内的仅限内部服务建立连接，还可以强制服务器连接到任意外部系统，从而泄露敏感数据 。

# 1. 针对服务器本身

攻击者诱使应用程序通过其 环回网络接口向托管应用程序的服务器发出 HTTP 请求，例如一个购物应用程序，它允许用户查看某项商品是否在特定商店中有库存，要提供库存信息，应用程序必须查询各种后端 API，具体取决于相关产品和商店，该功能通过前端 HTTP 请求将 URL 传递给相关的后端 API 端点来实现，查看库存会发出如下请求:

这会导致服务器指定的 URL 请求，检索库存状态并将其返回给用户。这里攻击者可以修改请求以指定服务器本身的本地 URL，获取 /admin URL 的内容将其返回给用户

出现这种隐式信任本地计算请求的原因：

访问控制检查可能在位于应用程序服务器前面的不同组件中实现，当与服务器本身建立连接时，会绕过检查；

出于灾难恢复的目的，为管理员提供一种在丢失凭据时恢复系统的办法；

管理界面可能正在监听与主应用程序不同的端口。

# 2. 针对其他后台系统

SSRF 经常出现的另一种类型的信任关系是应用程序服务器能够与用户无法直接访问的其他后端系统进行交互，这些系统具有不可路由的私有 IP 地址，由于后端系统受到网络拓扑的保护，它们具有较弱的安全态势，多数情况，内部后端系统包含敏感功能，任何能够与系统交互的人无需身份验证即可访问这些功能。

# 二。绕过常见的 SSRF 防御

# 1. 基于黑名单的输入过滤

一些应用程序会阻止包含主机名，如 127.0.0.1/localhost 或敏感 URL，如 /admin 的输入，

  1. 使用 127.0.0.1 的代替 IP 表示，如，2130706433、017700000001 或 127.1
  2. 注册自己的解析为 127.0.0.1 的域名，可以使用 burp
  3. 使用 URL 编码或大小写变体混淆被过滤的字符串

# 2. 基于白名单的输入过滤

一些应用程序只允许匹配、开头或包含允许值的白名单输入 ，可以利用 URL 解析中的不一致性来绕过，URL 规范中包含许多在实现 URL 的特定解析和验证时容易被忽视的特性：

  • 可以使用 @字符在主机名之前的 URL 中嵌入凭据， https://expected-host@evil-host ，会直接访问 evil-host

  • 可以使用 #字符来指示 URL 片段， https://evil-host#expected-host

  • 利用 DNS 命名层次结构将所需的输入放入自身控制的 DNS 域名中， https://expected-host.evil-host

  • 对字符进行 URL 编码以混淆 URL 解析代码，如果实现过滤的代码处理 URL 编码字符的方式与执行后端 HTTP 请求的代码不同，非常有用。

# URL 解析问题

上述特性都是因为 URL 解析和请求之间的不一致导致，验证 URL 困难的原因：

  • RFC3986 中的规范只是规范
  • WHATWG 定义了基于 RFC 的实现，但是不同语言有自己的实现

# RFC3986

1.URL 编码

对应字符超出允许的集合或用作分隔符时，百分比编码机制表示数据八位字节，由 “%” 后跟代表该八位字节数值的两个十六进制数字组成，如 %20 是二进制八位字节 “00100000” 的百分比编码，对应字符空格。URI 中的八位字节被百分比编码的唯一时间是在从其组成部分生成 URI 的过程中，生成后 URI 始终采用百分比编码形式。但是不是保留集中的字符相对应的百分比编码八位字节可以随时编码，百分号必须被编码为 %25，才能将该八位字节用作 URI 中的数据。

2.URI 构成

在 ssrf 中，协议部分只需关注 HTTP,HTTPS，查询和分段不需要关注，主要是在路径和权限。

权限部分由下一个斜杠，问号或 #或结尾终止， authority = [userinfo"@"]host[":"port]

# 3. 重定向绕过

允许 URL 的应用程序包含一个开放重定向漏洞，用于发出后端 HTTP 请求的 API 支持重定向，则可以构造一个满足过滤条件的 URL。

如应用程序包含一个开放重定向功能：

/product/nextProduct?currentProductId=6&path=http://evil-user.net

返回重定向到 http://evil-user.net

由于应用程序验证 stockAPI URL 是允许访问的域名，所以可以利用重定向进行构造

# 三。盲 SSRF 漏洞

检测盲 SSRF 最可靠的方法是使用 OAST 技术，而一般采用 DNS lookup 外带，但是简单识别触发外带 HTTP 请求的盲 SSRF 漏洞用处不大，可以对内部 IP 地址进行扫描，发送想检测漏洞的 payload。

也可以使用 SSRF 扫描器 - Collaborator Everywhere，ssrf 注入点在 get 或 post 的请求参数中存在，在 http 请求头中也存在，Collaborator Everywhere 给可能发发生 ssrf 的点都写入 payload，如果存在 ssrf，client 会检测到报告漏洞。

另一种方法是诱导应用程序连接到攻击者控制的系统，并向建立连接的 HTTP 客户端返回恶意响应，如果可以利用服务器 HTTP 实现中的客户端漏洞，就可以在应用程序基础架构中实现远程代码执行。

# 四.SSRF 漏洞的隐藏攻击面

# 1. 请求中部分 URL

有时应用程序仅将主机名或 URL 的一部分放入请求参数中，然后提交的值会在服务器端合并到请求的完整 URL 中，如果该值很容易识别为主机名或 URL 路径，潜在攻击面很明显，但是无法控制请求的整个 URL，会使 SSRF 利用受限。

# 2. 数据格式 URL

一些应用程序以其规范允许包含可能由数据解析器请求的格式的 URL 传输数据，使用最多是 XML 数据格式，将结构化数据从客户端传输到服务器，应用程序接受 XML 格式数据对其进行解析时很容易受到 XXE 攻击。

# 3. 通过 Referer 头的 SSRF

一些应用程序使用跟踪访问者的服务端分析软件，该软件经常在请求中记录 Referer 头，因为对跟踪传入链接特别有用，通常分析软件会访问出现在 Referer 头中的任何第三方 URL，用于分析引用站点内容，包括传入链接中使用的锚点。
