# 一.CSRF

实现 csrf 的三个关键条件：

  • 一个相关动作

应用程序中存在诱使攻击者有理由诱导的操作。

  • 基于 Cookie 的会话处理

应用程序仅依赖于 cookie 来识别发出请求的用户

  • 无不可预测的参数

手动创建 CSRF PoC 比较麻烦，可以利用 Burp 自动生成 --- 在需要测试的请求中右键，选择 Engagement tools/Generate CSRF PoC。

# 阻止 CSRF

# 1.Token

**token 要求：** 不可预测、绑定到用户会话、执行相关操作之前每种情况都需要经过严格验证。

Token 的生成：

应使用 PRNG (伪随机数生成器)，使用创建时的时间戳和静态密钥作为种子。

Token 的传输：

通常是在使用 POST 方法提交的 HTML 表单的隐藏字段中将 Token 传输到客户端， <input type="hidden" name="csrf-token" value="CIwNZNlR4XbisJF39I8yWnWX9wX4WFoz" /> , 为提高安全性包含 Token 的字段应尽早放置在 HTML 文档中，最好是在任何非隐藏字段之前及在 HTML 中嵌入用户可控数据的任何位置之前。Token 不应在 cookie 中传输

Token 的验证：

生成 Token 时，它应该存储在服务器的用户会话数据中，当接收到需要验证的后续请求时，服务端应用程序应验证请求是否包含与存储在用户会话中相匹配的 Token，无论请求的 HTYP 方法或内容类型如何都应该执行此验证。

# 2.SameSite Cookie

SameSite 属性可以让 Cookie 在跨站请求时不会被发送，从而可以阻止跨站请求伪造攻击（CSRF）。

# 属性值

SameSite 可以有下面三种值：

  1. Strict 仅允许一方请求携带 Cookie，即浏览器将只发送相同站点请求的 Cookie，即当前网页 URL 与请求目标 URL 完全一致。
  2. Lax 允许部分第三方请求携带 Cookie
  3. None 无论是否跨站都会发送 Cookie

之前默认是 None 的，Chrome80 后默认是 Lax。

# 跨域和跨站

首先要理解的一点就是跨站和跨域是不同的。同站 (same-site)/ 跨站 (cross-site)」和第一方 (first-party)/ 第三方 (third-party) 是等价的。但是与浏览器同源策略（SOP）中的「同源 (same-origin)/ 跨域 (cross-origin)」是完全不同的概念。

同源策略的同源是指两个 URL 的协议 / 主机名 / 端口一致。例如，https://www.taobao.com/pages/...，它的协议是 https，主机名是 www.taobao.com，端口是 443。

同源策略作为浏览器的安全基石，其「同源」判断是比较严格的，相对而言，Cookie 中的「同站」判断就比较宽松：只要两个 URL 的 eTLD+1 相同即可，不需要考虑协议和端口。其中，eTLD 表示有效顶级域名，注册于 Mozilla 维护的公共后缀列表（Public Suffix List）中，例如，.com、.co.uk、.github.io 等。eTLD+1 则表示，有效顶级域名 + 二级域名，例如 taobao.com 等。

举几个例子，www.taobao.com 和 www.baidu.com 是跨站，www.a.taobao.com 和 www.b.taobao.com 是同站，a.github.io 和 b.github.io 是跨站 (注意是跨站)。

# 改变

接下来看下从 None 改成 Lax 到底影响了哪些地方的 Cookies 的发送？直接来一个图表：

从上图可以看出，对大部分 web 应用而言，Post 表单，iframe，AJAX，Image 这四种情况从以前的跨站会发送三方 Cookie，变成了不发送。

Post 表单：应该的，学 CSRF 总会举表单的例子。

iframe：iframe 嵌入的 web 应用有很多是跨站的，都会受到影响。

AJAX：可能会影响部分前端取值的行为和结果。

Image：图片一般放 CDN，大部分情况不需要 Cookie，故影响有限。但如果引用了需要鉴权的图片，可能会受到影响。

除了这些还有 script 的方式，这种方式也不会发送 Cookie，像淘宝的大部分请求都是 jsonp，如果涉及到跨站也有可能会被影响。

# 二。常见的 CSRF 漏洞

# 1.Token 验证取决于请求方法

POST 方法要求验证 Token，但是 GET 方法不验证造成绕过，

# 2.Token 验证取决于是否存在

某些应用在 Token 存在时正确验证，但如果省略了 Token 则跳过了验证，可以通过删除包含 Token 的整个参数来绕过。

# 3.Token 和用户会话无关

某些应用不会验证令牌与发出请求的用户是否属于同一会话，相反它维护已发布的全局 Token 池，并接受该池中出现的任何 Token，攻击者可以使用自己的账户登录应用，获取有效 Token 之后，再将该 Token 提供给受害用户。

# 4.Token 在 Cookie 中复制

一些应用不维护已颁发的 Token 服务端记录，而是在 cookie 和请求参数中复制每个 Token，验证后续请求时，应用只需验证请求参数中提交的 Token 与 cookie 中提交的值是否匹配 --- 易于实现且避免服务端状态的需要，在这种情况下如果网站包含任何 cookie 设置功能，攻击者可以再次执行 CSRF 攻击，无需获取自己的有效令牌，利用 cookie 设置行为将假 Token 放入受害者浏览器即可。

# 5. 基于 Referer 头的 CSRF 防御

一些应用使用 Referer 头来防御 CSRF，验证请求是否来自应用自己的域，如果该标头被省略，则会跳过验证。

最简单删除 Referer 头的方式: <meta name="referrer" content="never">

某些应用验证 Rerferer 头中的域名以期望值开头，那么攻击者可以将其作为自己域的子域： http://vulnerable-website.com.attacker-website.com/csrf-attack , 同样如果只是验证 Referer 是否只包含自己的域名，那么攻击者可以将所需的值放在 URL 的其他位置：

http://attacker-website.com/csrf-attack?vulnerable-website.com

现在浏览器默认从 Referer 头中去除查询字符串，可以通过设置 Referrer-Policy: unsafe-url 来覆盖此行为，以确保发送包括查询字符串的 URL。
