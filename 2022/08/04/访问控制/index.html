# 一。访问控制

访问控制依赖于身份验证和会话管理：

  • 身份验证识别用户并确认他们就是他们所说的人。

  • 会话管理识别同一用户正在发出哪些后续 HTTP 请求。

  • 访问控制确定是否允许用户执行他们正在尝试执行的操作。

访问控制分为：

  • 垂直访问控制

限制对其他类型用户不可用敏感功能的访问机制，不同类型的用户可以访问不同的应用程序功能，管理员可以修改任何用户的账户，普通用户则无权。

  • 水平访问控制

将资源的访问权限控制在特定允许访问这些资源的用户，不同用户可以访问同一类型的资源子集。银行应用程序允许用户查看交易并从他们自己的账户进行支付，但不允许任何其他用户的账户。

  • 上下文相关的访问控制

根据应用程序的状态或用户与应用程序的交互来限制对功能和资源的访问，可防止用户以错误的顺序执行操作。零售网站会阻止用户在付款后修改其购物车的内容。

# 访问控制安全模型

# 1. 程序访问控制

使用编程访问控制，用户权限矩阵存储在数据库，并且访问控制参考该矩阵以编程方式应用。

# 2. 自主访问控制 / DAC

通过自主访问控制，对资源或功能的访问受到用户或指定用户组的限制，资源或功能的所有者能够向用户委派或分配访问权限。

# 3. 强制访问控制 / MAC

一种集中控制的访问控制系统，其中主体对某些对象 (文件或其他资源) 的访问受到限制，资源的用户和所有者没有能力委派或修改其资源的访问权限。

# 4. 基于角色的访问控制 / RBAC

可以定义命名角色，并为其分配访问控制权限，然后将用户分配给单个或多个角色。采购员可能被定义为对采购分类帐功能和资源的子集具有访问权限的角色。随着员工离开或加入组织，访问控制管理被简化为定义或撤销采购员角色的成员资格。

# 三。越权漏洞

# 1. 水平越权

在某些应用中，可利用的参数的值无法预测，可能使用全局唯一标识符来标识用户，而不是递增数字，属于其他用户的全局标识符可能会在引用用户的应用程序的地方公开 - 用户消息或评论。

在某些情况下，应用程序会检测到不允许用户访问资源，并返回一个重定向到登陆页面，但是包含重定向的响应可能仍然包含属于目标用户的一些敏感数据。

任意修改其他用户密码 + 可修改管理员密码 = 水平越权 + 垂直越权

# 2. 不安全的直接对象引用 / IDOR

应用程序使用用户的输入直接访问对象并且攻击者可以修改输入以获得未授权的访问时，就是 IDOR。

直接引用数据库对象：

https://insecure-website.com/customer_account?customer_number=132355

直接使用 customer_number 值在数据库中查询。

直接引用静态文件：

https://insecure-website.com/static/12144.txt

# 3. 多步骤访问控制漏洞

网站通过一系列步骤实现重要功能，比如更新用户详细信息可能涉及以下步骤：

  • 加载包含特定用户详细信息的表单
  • 提交更改
  • 查看更改并确认

有时网站会对其中一些步骤实施严格的访问控制，而忽略其他步骤，假设访问控制正确应用于第一步和第二步，但没有应用于第三步，攻击者可以用过跳过前两个步骤直接提交带有所需参数的第三步请求来获得对该功能的未经授权的访问。

# 4. 基于 Referer 的访问控制

一些网站基于 HTTP 请求中的 Referer 请求头进行访问控制。Referer 头通常指示发起请求的页面。

假设一个应用程序对 /admin 的主管理页面强制实施访问控制。但对于 /admin/deleteUser 等子页面仅检查 Referer 头，如果 Referer 头中包含 /admin URL，则允许该请求。

# 5. 基于位置的访问控制

可以利用代理、VPN 或更改客户端定位来绕过。

# 6. 垂直越权

# 1) 未受保护的功能

当应用程序不对敏感功能实施任何保护时，就会出现垂直越权。例如，管理功能可能从管理员的欢迎界面链接，而不是从用户的欢迎页面链接，但是用户可以通过直接浏览相关的 URL 来访问管理功能（可以尝试目录扫描发现敏感链接）。

# 2) 基于参数的访问方法

一些应用程序在登录时确定用户的访问权限或角色，然后将此信息存储在用户可控制的位置，例如隐藏字段，cookie 或预设的查询字符串参数。

# 3) 配置错误导致访问控制中断

1. 一些应用程序基于用户角色限制对特定 URL 和 HTTP 方法的访问来在平台层实施访问控制，例如，

DENY: POST, /admin/deleteUser, managers , 该规则会拒绝 managers 组中的用户访问 /admin/deleteUser 上的 POST 方法，但是一些框架支持各种非标准的 HTTP 头，可用于覆盖原始请求中的 URL， X-Original-URL 和 X-Rewrite-URL , 网站使用严格的前端控制来限制基于 URL 的访问，但应用程序允许通过请求头覆盖 URL，则可以绕过访问控制。

2. 前端控件根据 URL 和 HTTP 方法限制访问，某些网站在执行操作时可以容忍其他 HTTP 请求方法，如果攻击者可以使用 GET 或其他请求方法对受限 URL 执行操作，则可以绕过。

# 三。防止访问控制漏洞

  • 不能依靠混淆来进行访问控制
  • 除非资源旨在公开访问，否则默认拒绝访问
  • 代码级别，强制开发人员声明每个资源允许的访问权限，默认拒绝访问
  • 彻底审核和测试访问机制
