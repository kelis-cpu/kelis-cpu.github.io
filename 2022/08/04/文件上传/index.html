# 一.web 服务器如何处理静态文件请求

历史上，网站都是由静态文件组成，这些文件会在用户请求时提供给用户，每个请求的路径可以与服务器文件系统上的目录和文件层次结构 1:1 映射，现在网站越来越动态，请求路径与文件系统无直接关系，但处理这些静态文件的过程大致相同，在某些时候，服务器会解析请求中的路径以识别文件扩展名，然后使用它来确定所请求文件的类型，通常通过将其与扩展名和 MIME 类型之间的预配置映射列表进行比较，接下来取决于文件类型和服务器的配置：

如果文件不可执行，服务器在 HTTP 响应中将文件的内容发送给客户端。

如果文件可执行，如 PHP 文件，并且服务器被配置为执行这种类型的文件，它会在运行脚本之前根据 HTTP 请求头和参数分配变量，然后将文件执行结果以 HTTP 响应形式发送到客户端；如果服务器未配置这种类型文件可执行，通常会响应错误，文件内容可能以纯文本的形式提供给客户端。

# 二。利用有缺陷的文件上传验证机制

# 1. 文件类型验证

消息正文被拆分为每个表单输入的单独部分，其中每个部分都有自己的 Content-Type 头，它告诉服务器使用此输入提交的数据的 MIME 类型。网站可能会尝试验证文件上传的一种方法是检查此输入的 Content-Type 头与预期的 MIME 类型匹配，如果不进一步检查文件内容是否实际匹配假定的 MIME 类型，则可以使用 Burp 绕过。

# 2. 防止在用户可访问目录执行文件

将 webshell 上传到不该包含用户提供的文件的不同目录。服务器最终可能会执行上传的脚本。Web 服务器通常使用 multiprt/form-data 请求中的文件名字段来确定文件应该保存的名称和位置，可以通过目录穿越来进行绕过。

即使可以将所有请求发送到一个域名，但它通常执行反向代理服务器 - 负载均衡器，请求通常由其他其它服务器处理，这些服务器也可能配置不同。

# 3. 危险文件类型的黑名单不足

黑名单的做法本质是有缺陷的，很难明确阻止每个可能用于执行代码的文件扩展名，可以使用鲜为人知的替代文件扩展名绕过黑名单，比如 .php 可以使用 .php5,.shtml 进行替代。

# 4. 覆盖服务器配置

服务器通常不会执行文件，除非已被配置为可执行该类型文件，如在 Apache 服务器执行客户端请求之前，开发人员可能在配置文件 /etc/apache2/apache2.conf 中配置

许多服务器还允许开发人员在各个目录中创建特殊的配置文件，以便覆盖或添加一个或多个全局设置，如 Apache 服务器将从 .htaccess 文件中加载特定于目录的配置；IIS 服务器从 web.config 进行特定目录的配置。

# 5. 混淆文件扩展名

最详尽的黑名单也可能被混淆技术绕过，假设验证代码区分大小写并且无法识别 .pHp 实际是一个 .php 文件，随后将文件扩展名映射到 MIME 类型的代码不区分大小写，这种差异就能绕过验证，最终文件被服务器执行。进行绕过的其他方法：

  • 提供多个扩展名，根据解析文件名的算法，以下 ex.php.jpg 文件可能被解析为 php 文件或 jpg 文件。
  • 扩展名末端添加空格或点， ex.php. ，一些组件会移除或忽略。
  • 对点、正反斜杠进行 URL 编码或双 URL 编码，验证文件扩展名时该值未被解码，但在服务器端被解码。
  • 在文件扩展名前添加分号或 URL 编码的空字节字符。如果验证使用 PHP 或 Java 编写，服务器使用 C/C++ 处理文件，会导致文件名结尾出现差异： ex.asp;.jpg或ex.asp%00.jpg 。
  • 尝试使用多字节 unicode 字符，在 unicode 转换或 规范化后可能会转换为空字节或点，文件名被解析为 UTF-8 字符串，像 xC0x2E、xC4xAE 或 xC0xAE 之类的序列可能会被转化为 x2E，用于路径之前会转换为 ASCII 字符。
  • 如果不采用递归删除扩展名，可能会双写绕过。

# 6. 有缺陷的文件内容验证

在图像上传功能的情况下，服务器可能会尝试验证图像的内在属性，如尺寸，某些文件类型可能总是在页眉或页脚中包含特定的字符序列，这些可以用作指纹或签名来确定内容是否与预期的类型匹配，例如 jpeg 总是以 FF D8 FF 开头。

# 7. 文件上传条件竞争

现在网站框架不会将文件直接上传到预期目的地，而是首先上传临时的沙箱目录并且随机命名以覆盖现有的文件，并且在通过验证之后再将其传输到目的地。但是一些网站将文件直接上传到主文件系统，如果没有通过验证，则再次将其删除，依赖于病毒软件会需要几毫秒时间，但是也可以被攻击者利用， 使用 Turbo Intruder 扩展。

基于 URL 的文件上传中的条件竞争

在通过 URL 上传文件的函数中可能会出现条件竞争，在这种情况下，服务器必须通过 Internet 获取文件并创建副本然后才能执行验证，由于文件是 HTTP 加载的，开发人员无法使用其框架的内置机制来安全验证文件，只能手动创建自己的流程来临时存储和验证文件。为了使攻击成功，可以上传较大的文件来延长文件的处理时间，如果以块的形式处理文件，可以开始时创建一个带有有效负载的恶意文件，然后是大量的任意填充字节。

# 防止文件上传漏洞

  • 采用允许扩展名的白名单
  • 确保文件名不包含任何可能被解释为目录或遍历序列 (../) 的子字符串
  • 重命名上传的文件
  • 在文件完成验证之前不要上传到服务器的永久文件系统
  • 尽可能使用已建立的框架来预处理文件上传，而不是尝试编写自己的验证机制
